# Deployment Guide

This document covers:
- first-time deployment to an Ubuntu server
- updating the deployed service after pushing new code
- the correct model and endpoint combinations for this project

## Server assumptions

- OS: Ubuntu
- App directory: `/home/ubuntu/ai-practitioner-blog`
- Service name: `ai-practitioner-blog`
- Reverse proxy: Nginx
- App port: `3000`

## 1. First-time deployment

### 1.1 Prepare the server

Make sure your cloud security group allows:
- `22` for SSH
- `80` for HTTP
- `443` for HTTPS if you plan to add SSL later

Log in to the server:

```bash
ssh ubuntu@YOUR_SERVER_IP
```

### 1.2 Clone the repository

```bash
cd /home/ubuntu
git clone https://github.com/yjg-djb/ai-practitioner-blog.git
cd ai-practitioner-blog
```

### 1.3 Create production env

```bash
cp .env.example .env.production
nano .env.production
```

Fill `.env.production` according to one of the following modes.

### 1.4 Choose the correct upstream configuration

#### Option A: DashScope coding endpoint

Use this only with coder models.

```env
HOST="0.0.0.0"
PORT="3000"
APP_URL="http://YOUR_SERVER_IP"

OPENAI_COMPAT_BASE_URL="https://coding.dashscope.aliyuncs.com"
OPENAI_COMPAT_API_KEY="YOUR_KEY"

VITE_OPENAI_API_BASE="/api/openai"
VITE_OPENAI_MODEL="qwen3-coder-plus"
VITE_OPENAI_AVAILABLE_MODELS="qwen3-coder-plus,qwen3-coder-next"
```

Important:
- `qwen3.5-plus` should not be used with `https://coding.dashscope.aliyuncs.com`
- if you use the coding endpoint with a general model, the request may hang or fail

#### Option B: DashScope general OpenAI-compatible endpoint

Use this for general models such as `qwen3.5-plus`.

```env
HOST="0.0.0.0"
PORT="3000"
APP_URL="http://YOUR_SERVER_IP"

OPENAI_COMPAT_BASE_URL="https://dashscope.aliyuncs.com/compatible-mode"
OPENAI_COMPAT_API_KEY="YOUR_MODEL_STUDIO_KEY"

VITE_OPENAI_API_BASE="/api/openai"
VITE_OPENAI_MODEL="qwen3.5-plus"
VITE_OPENAI_AVAILABLE_MODELS="qwen3.5-plus,qwen3-max-2026-01-23,glm-5,glm-4.7,kimi-k2.5,MiniMax-M2.5"
```

Important:
- the project server automatically appends `/v1`
- do not write `/v1` into `OPENAI_COMPAT_BASE_URL`
- the API key must match the selected endpoint

### 1.5 Install and start the service

```bash
bash deploy/install-ubuntu.sh
```

This script will:
- install Node.js 22 and Nginx if needed
- run `npm ci`
- build the frontend
- create and enable the `systemd` service
- configure Nginx to proxy to the Node app
- disable Nginx buffering so streaming responses work correctly

### 1.6 Verify deployment

```bash
curl http://127.0.0.1:3000/health
sudo systemctl status ai-practitioner-blog --no-pager
sudo nginx -t
```

Then open:

```text
http://YOUR_SERVER_IP
```

## 2. Update deployment after code changes

When you have already pushed new code to GitHub, update the server with:

```bash
cd /home/ubuntu/ai-practitioner-blog
git pull
npm ci
npm run build
sudo systemctl restart ai-practitioner-blog
sudo systemctl reload nginx
```

If `package.json` did not change, you can use:

```bash
cd /home/ubuntu/ai-practitioner-blog
git pull
npm run build
sudo systemctl restart ai-practitioner-blog
sudo systemctl reload nginx
```

## 3. Recommended release check after each update

Run these checks after every deployment:

```bash
curl http://127.0.0.1:3000/health
sudo systemctl status ai-practitioner-blog --no-pager
```

Then verify in the browser:
- page opens normally
- title is correct
- AI assistant can reply
- streaming output appears progressively instead of waiting for the full answer

## 4. Useful operations

### View service logs

```bash
sudo journalctl -u ai-practitioner-blog -n 100 --no-pager
```

### Restart service

```bash
sudo systemctl restart ai-practitioner-blog
```

### Reload Nginx

```bash
sudo systemctl reload nginx
```

### Check which process is listening on port 3000

```bash
sudo ss -ltnp | grep 3000
```

## 5. Troubleshooting

### Page opens but AI assistant stays in the loading state

Check these items in order:
- `.env.production` has the correct `OPENAI_COMPAT_BASE_URL`
- `.env.production` has a valid `OPENAI_COMPAT_API_KEY`
- the model matches the endpoint type
- `curl http://127.0.0.1:3000/health` returns `ok`
- `sudo journalctl -u ai-practitioner-blog -n 100 --no-pager` has no upstream errors

### Streaming works locally but not on the server

Check:
- Nginx config was regenerated by `bash deploy/install-ubuntu.sh`
- Nginx buffering is disabled
- the request path is still `/api/openai/chat/completions`

### After update, the page still shows old content

Run:

```bash
git pull
npm run build
sudo systemctl restart ai-practitioner-blog
sudo systemctl reload nginx
```

Then hard-refresh the browser.
